# Adaptive Deep Research -- Specification Addendum

> **Source brief**: `ideas/002-adaptive-deep-research.md`
> **Parent spec**: `specs/autonomous-deep-research.spec.md` (v1.1)
> **Feature branch**: `002-adaptive-deep-research`
> **Status**: Draft
> **Version**: 1.0

---

## 1. Overview

This specification addendum redesigns the `DeepResearchOrchestrator` internals from a pre-generated query fan-out strategy into an adaptive reasoning loop. The current implementation generates all query variants upfront (one LLM call) and executes them independently with no inter-round feedback. The new design mirrors a human researcher's workflow: plan an initial search strategy, execute it, analyze the results for knowledge gaps, reason about what to search next, and repeat until the topic is comprehensively covered or configured limits are reached.

This addendum modifies only the internal behavior of `DeepResearchOrchestrator` (Section 4.3 of the parent spec). All external contracts -- state key format (`research_{idx}_{provider}`), pipeline position, configuration schema, and downstream agent expectations -- remain unchanged or are backward-compatible extensions.

**Supersedes**: Parent spec Section 4.3 (FR-MRR-001 through FR-MRR-011), Section 4.3's implementation contract, Section 8.3 (QueryExpanderAgent and DeepResearchOrchestrator contracts), and related test requirements in Section 11.

---

## 2. Goals & Success Criteria

- **SC-ADR-001**: Deep-mode topics SHALL produce research results that cover at least 3 of the 5 key aspects identified by the planning phase (measured via the analysis agent's coverage assessment in final round).
- **SC-ADR-002**: The adaptive loop SHALL exit early (before `max_research_rounds`) in at least 30% of topic-provider executions when saturation is detected, reducing unnecessary API calls compared to the fixed fan-out approach.
- **SC-ADR-003**: Each search round's query SHALL demonstrably differ from the previous round's query (no duplicate queries across rounds within the same topic-provider execution).
- **SC-ADR-004**: The reasoning chain SHALL be logged at INFO level, providing full transparency into the plan -> search -> analyze -> decide cycle for debugging and quality assessment.
- **SC-ADR-005**: All existing tests (parent spec SC-006) SHALL continue to pass. Standard-mode topics SHALL produce identical behavior.
- **SC-ADR-006**: Deep-mode research with 3 rounds SHALL complete within 3-5 minutes per topic-provider (including planning and analysis overhead).

- **Recipient**: Receives newsletter. Indirectly benefits from deeper, more comprehensive topic coverage.

## 4. Functional Requirements

### 4.1 Adaptive Research Loop Architecture

- **FR-ADR-001**: The `DeepResearchOrchestrator` SHALL replace the pre-generated query fan-out with an adaptive reasoning loop consisting of four phases per round: Plan (round 0 only), Search, Analyze, Decide.
- **FR-ADR-002**: The adaptive loop SHALL follow this sequence for each deep-mode topic-provider combination:
  1. **Plan**: Invoke `PlanningAgent` (LlmAgent) to interpret the query, identify key aspects, and produce the initial search query. Runs once at the start.
  2. **Search**: Invoke `DeepSearchRound` (LlmAgent, existing) with the current query. Produces search results.
  3. **Analyze**: Invoke `AnalysisAgent` (LlmAgent) with accumulated context and latest results. Evaluates findings, identifies knowledge gaps, determines saturation, and suggests next query.
  4. **Decide**: Orchestrator (Python logic) checks exit criteria. If not exiting, extracts the next query from analysis output and returns to step 2.
- **FR-ADR-003**: The `QueryExpanderAgent` (parent spec FR-MRR-004) SHALL be removed. Query generation is now performed adaptively by the `AnalysisAgent` at each round, informed by prior results.
- **FR-ADR-004**: The `DeepResearchOrchestrator` SHALL remain a `BaseAgent` subclass (parent spec FR-MRR-001). Only the internal execution logic within `_run_async_impl` changes.
- **FR-ADR-005**: Round 0 SHALL use the initial search query produced by the `PlanningAgent`. If the `PlanningAgent` returns invalid output, the original topic query SHALL be used as fallback.
- **FR-ADR-006**: Rounds 1 through N SHALL use the `next_query` produced by the `AnalysisAgent` from the previous round. If the `AnalysisAgent` returns invalid output, a fallback query SHALL be generated by appending a gap-oriented suffix from a predefined list to the original query.

**Implementation Contract -- Adaptive Loop**:

Per deep-mode topic, per provider, the agent behavior:
```
DeepResearchOrchestrator_{idx}_{provider} (BaseAgent)
    # Phase 0: Planning (once)
    #   1. Create/invoke PlanningAgent (LlmAgent) -> initial plan + first query
    #
    # Phase 1-N: Adaptive Loop (up to max_research_rounds)
    #   2. Create/invoke DeepSearchRound (LlmAgent) -> search results
    #   3. Create/invoke AnalysisAgent (LlmAgent) -> gap analysis + next query
    #   4. Check exit criteria (saturation, max rounds, max searches)
    #   5. If continuing: use next_query from analysis, go to step 2
    #
    # Phase Final: Merge
    #   6. Merge all round results into research_{idx}_{provider}
    #   7. Cleanup intermediate state keys
```

### 4.2 Initial Planning Phase

- **FR-ADR-010**: The `PlanningAgent` SHALL be an `LlmAgent` sub-agent created by the orchestrator using the research model (`gemini-2.5-flash`).
- **FR-ADR-011**: The `PlanningAgent` SHALL receive the original query and topic name as input via its instruction prompt.
- **FR-ADR-012**: The `PlanningAgent` SHALL output a JSON object to the state key `adaptive_plan_{idx}_{provider}` containing:
  - `query_intent` (string, 1-200 chars): One sentence describing what the user wants to learn.
  - `key_aspects` (list of 3-5 strings, each 1-100 chars): Distinct aspects/angles to investigate.
  - `initial_search_query` (string, 1-500 chars): The first search query to execute.
  - `search_rationale` (string, 1-200 chars): Why this query is the best starting point.
- **FR-ADR-013**: If the `PlanningAgent` returns invalid JSON or any required field is missing, the orchestrator SHALL:
  - Use the original topic query as `initial_search_query`.
  - Generate default `key_aspects` from a predefined list: `["recent developments", "expert opinions", "data and statistics", "industry implications", "emerging trends"]`.
  - Log a warning: `"[AdaptiveResearch] Planning failed for {topic_name}/{provider}, using fallback"`.
- **FR-ADR-014**: The `PlanningAgent` SHALL NOT have access to search tools. It operates purely on the LLM's knowledge to plan the research strategy.

**PlanningAgent instruction contract**:

```
You are a research strategist. Analyze the following research query and plan
a systematic search strategy.

Topic: {topic_name}
Query: {query}

Your task:
1. Understand the core intent of this research query.
2. Identify 3-5 distinct aspects or angles that a thorough investigation should cover.
3. Determine the single best initial search query to start with.

Produce a JSON object with exactly these fields:
- "query_intent": one sentence describing what the user wants to learn
- "key_aspects": list of 3-5 distinct aspects to investigate (strings)
- "initial_search_query": the first search query to execute (string)
- "search_rationale": one sentence explaining why this query is the best starting point

Output ONLY the JSON object. No other text.
```

**PlanningAgent output schema**:

| Field | Type | Required | Constraints | Description |
|-------|------|----------|-------------|-------------|
| `query_intent` | `string` | Yes | 1-200 chars | Core intent of the research query |
| `key_aspects` | `list[string]` | Yes | 3-5 items, each 1-100 chars | Distinct investigation angles |
| `initial_search_query` | `string` | Yes | 1-500 chars | First search query |
| `search_rationale` | `string` | Yes | 1-200 chars | Why this starting point |

### 4.3 Search Phase (Modified)

- **FR-ADR-020**: The `DeepSearchRound` LlmAgent sub-agent SHALL remain functionally unchanged from the parent spec (FR-MRR-005, FR-MRR-009, FR-MRR-010). It executes a single search with the given query and writes results to `deep_research_latest_{idx}_{provider}`.
- **FR-ADR-021**: The search query for each round SHALL be provided by the adaptive loop (from PlanningAgent for round 0, from AnalysisAgent for subsequent rounds) rather than from a pre-generated variant list.
- **FR-ADR-022**: The `DeepSearchRound` instruction prompt (from `research_google.py` and `research_perplexity.py`) SHALL remain unchanged. Only the query string passed to the prompt changes.

### 4.4 Analysis Phase (New)

- **FR-ADR-030**: After each search round, the orchestrator SHALL invoke an `AnalysisAgent` (LlmAgent) to evaluate the accumulated research and determine the next action.
- **FR-ADR-031**: The `AnalysisAgent` SHALL receive the following context via its instruction prompt:
  - The original query and topic name.
  - The `key_aspects` from the planning phase.
  - A condensed summary of all prior rounds (from the accumulated `AdaptiveContext`).
  - The full text of the latest search round's results.
  - The current round index and remaining search budget.
- **FR-ADR-032**: The `AnalysisAgent` SHALL output a JSON object to the state key `adaptive_analysis_{idx}_{provider}` containing:
  - `findings_summary` (string, 1-500 chars): Brief summary of new information from the latest round.
  - `knowledge_gaps` (list of 0-5 strings, each 1-200 chars): Remaining unanswered questions or uncovered aspects.
  - `coverage_assessment` (string, 1-200 chars): One sentence describing how comprehensively the topic has been covered so far.
  - `saturated` (boolean): `true` if further searching would yield diminishing returns; `false` otherwise.
  - `next_query` (string or null, 1-500 chars when present): The next search query to fill the most important gap. `null` when `saturated` is `true`.
  - `next_query_rationale` (string or null, 1-200 chars when present): Why this query addresses the most critical gap. `null` when `saturated` is `true`.
- **FR-ADR-033**: If the `AnalysisAgent` returns invalid JSON, the orchestrator SHALL:
  - Set `saturated` to `false`.
  - Generate a fallback `next_query` by appending the next unused suffix from the predefined fallback list to the original query.
  - Log a warning: `"[AdaptiveResearch] Analysis failed for {topic_name}/{provider} round {N}, using fallback query"`.
- **FR-ADR-034**: The `AnalysisAgent` SHALL NOT have access to search tools. It operates purely on the provided context to perform analysis and reasoning.
- **FR-ADR-035**: The `AnalysisAgent` SHALL use the research model (`gemini-2.5-flash`).

**AnalysisAgent instruction contract**:

```
You are a research analyst evaluating search results for the topic "{topic_name}".

Original research goal: {query}

Key aspects to cover:
{key_aspects_formatted}

Research accumulated so far (prior rounds):
{prior_rounds_summary}

Latest search results (Round {round_idx}, query: "{current_query}"):
{latest_results}

Remaining search budget: {remaining_searches} searches

Your task:
1. Summarize the key NEW information found in the latest search results.
2. Compare what has been found against the key aspects. What gaps remain?
3. Assess whether further searching would add meaningful new information.
4. If not saturated, suggest the single most important next search query to fill the largest gap.

Produce a JSON object with exactly these fields:
- "findings_summary": brief summary of new findings from this round (string)
- "knowledge_gaps": list of remaining gaps or unanswered questions (0-5 strings, empty list if none)
- "coverage_assessment": one sentence describing current coverage completeness (string)
- "saturated": true if further searching would yield diminishing returns, false otherwise (boolean)
- "next_query": the next search query to fill the most important gap, or null if saturated (string or null)
- "next_query_rationale": why this query addresses the most critical gap, or null if saturated (string or null)

Saturation guidelines:
- Set saturated=true if: all key aspects are well-covered, OR the latest round added no significant new information, OR remaining gaps are too narrow for web search to resolve.
- Set saturated=false if: major aspects remain uncovered, OR important contradictions need resolution, OR key data points are missing.

Output ONLY the JSON object. No other text.
```

**AnalysisAgent output schema**:

| Field | Type | Required | Constraints | Description |
|-------|------|----------|-------------|-------------|
| `findings_summary` | `string` | Yes | 1-500 chars | New findings from latest round |
| `knowledge_gaps` | `list[string]` | Yes | 0-5 items, each 1-200 chars | Remaining gaps |
| `coverage_assessment` | `string` | Yes | 1-200 chars | Coverage completeness statement |
| `saturated` | `boolean` | Yes | - | Whether further search would add value |
| `next_query` | `string` or `null` | Yes | 1-500 chars when present | Next search query |
| `next_query_rationale` | `string` or `null` | Yes | 1-200 chars when present | Rationale for next query |

### 4.5 Saturation Detection & Exit Criteria

- **FR-ADR-040**: The orchestrator SHALL exit the adaptive loop when ANY of the following conditions is met:
  1. The `AnalysisAgent` sets `saturated: true` AND the orchestrator has completed at least `min_research_rounds` rounds (configurable safety minimum, default 2).
  2. The `AnalysisAgent` returns an empty `knowledge_gaps` list (all aspects covered).
  3. The current round index reaches `max_research_rounds` (hard cap from config).
  4. The total number of search API calls reaches `max_searches_per_topic` (hard cap from config).
- **FR-ADR-041**: The 15-URL threshold early exit (parent spec FR-MRR-007) SHALL be removed. Saturation is now determined by LLM-driven analysis, not URL counting.
- **FR-ADR-042**: If the `AnalysisAgent` reports `saturated: true` before `min_research_rounds` rounds have completed, the orchestrator SHALL override the saturation signal and continue. This prevents premature exit on queries with sparse initial results.
- **FR-ADR-043**: When early exit occurs due to saturation, the orchestrator SHALL log: `"[AdaptiveResearch] Topic {name}/{provider}: saturated at round {N} - {coverage_assessment}"` at INFO level.
- **FR-ADR-044**: When the loop exits at the hard cap without saturation, the orchestrator SHALL log: `"[AdaptiveResearch] Topic {name}/{provider}: reached max rounds ({N}) without saturation. Gaps remaining: {gaps}"` at WARNING level.

### 4.6 State Management for Reasoning Chain

- **FR-ADR-050**: The orchestrator SHALL maintain an `AdaptiveContext` data structure in session state at key `adaptive_context_{idx}_{provider}` that accumulates across rounds. This structure tracks the full reasoning chain.
- **FR-ADR-051**: The `AdaptiveContext` SHALL contain:
  - `plan` (object): The output of the `PlanningAgent` (or fallback values).
  - `rounds` (list of objects): One entry per completed round, each containing:
    - `round_idx` (int): The round number (0-based).
    - `query` (string): The search query used.
    - `findings_summary` (string): From the `AnalysisAgent` output.
    - `knowledge_gaps` (list of strings): From the `AnalysisAgent` output.
    - `urls_found` (int): Number of new unique URLs found in this round.
    - `coverage_assessment` (string): From the `AnalysisAgent` output.
    - `saturated` (boolean): From the `AnalysisAgent` output.
- **FR-ADR-052**: The `AdaptiveContext` SHALL be passed to the `AnalysisAgent` as the `prior_rounds_summary` in its instruction. The orchestrator SHALL format the `rounds` list into a human-readable summary for the prompt.
- **FR-ADR-053**: All intermediate state keys (`adaptive_plan_{idx}_{provider}`, `adaptive_analysis_{idx}_{provider}`, `adaptive_context_{idx}_{provider}`, `deep_research_latest_{idx}_{provider}`, `research_{idx}_{provider}_round_{N}`) SHALL be cleaned up after the final merge, consistent with parent spec behavior.
- **FR-ADR-054**: The state key `deep_queries_{idx}_{provider}` (parent spec) SHALL no longer be written since upfront query expansion is removed.
- **FR-ADR-055**: The `AdaptiveContext` reasoning chain SHALL be persisted to a dedicated state key `adaptive_reasoning_chain_{idx}_{provider}` BEFORE intermediate key cleanup. This key SHALL NOT be cleaned up after merge, enabling post-run quality analysis and debugging. The persisted value SHALL be the full `AdaptiveContext` dict serialized as JSON.

### 4.7 Configuration Changes

- **FR-ADR-060**: The existing `max_research_rounds` field in `AppSettings` SHALL be reused without modification. Its semantic meaning changes from "max pre-generated query rounds" to "max adaptive reasoning rounds." Range: 1-5, default: 3.
- **FR-ADR-061**: A new optional field `max_searches_per_topic` SHALL be added to `AppSettings`:
  - Type: `int`
  - Default: value of `max_research_rounds` (dynamic default via model_validator)
  - Constraints: minimum 1, maximum 15
  - Description: Maximum total search API calls per topic-provider combination.
- **FR-ADR-062**: If `max_searches_per_topic` is less than `max_research_rounds`, the search budget SHALL be the binding constraint (the loop exits when searches are exhausted even if reasoning rounds remain).
- **FR-ADR-063**: When `max_research_rounds` is 1, the adaptive loop SHALL execute exactly one search with no analysis phase -- equivalent to current single-round deep research behavior (backward compatibility).
- **FR-ADR-064**: A new optional field `min_research_rounds` SHALL be added to `AppSettings`:
  - Type: `int`
  - Default: 2
  - Constraints: minimum 1, maximum 3, must be <= `max_research_rounds`
  - Description: Minimum rounds before saturation can trigger early exit. Replaces the hard-coded safety minimum of 2.
- **FR-ADR-065**: If `min_research_rounds` > `max_research_rounds`, config validation SHALL raise `ConfigValidationError` with a descriptive message.

**Implementation Contract -- Config Field**:

```python
# In AppSettings (newsletter_agent/config/schema.py)
max_research_rounds: int = Field(default=3, ge=1, le=5)
max_searches_per_topic: int = Field(default=None, ge=1, le=15)
min_research_rounds: int = Field(default=2, ge=1, le=3)

@model_validator(mode="after")
def resolve_adaptive_defaults(self) -> AppSettings:
    if self.max_searches_per_topic is None:
        self.max_searches_per_topic = self.max_research_rounds
    if self.min_research_rounds > self.max_research_rounds:
        raise ValueError(
            f"min_research_rounds ({self.min_research_rounds}) must be "
            f"<= max_research_rounds ({self.max_research_rounds})"
        )
    return self
```

YAML example:
```yaml
settings:
  max_research_rounds: 3      # max adaptive reasoning rounds (1-5)
  max_searches_per_topic: 5   # max search API calls per topic/provider (1-15, optional)
  min_research_rounds: 2      # min rounds before saturation exit (1-3, optional, default 2)
```

YAML example (backward compatible, no new field):
```yaml
settings:
  max_research_rounds: 3      # defaults max_searches_per_topic to 3, min_research_rounds to 2
```

## 5. User Stories

### US-ADR-01 -- Adaptive Research Improves Coverage (Priority: P1) MVP

**As an** Operator, **I want** deep-mode research to adaptively choose each search query based on what previous rounds found and what gaps remain, **so that** the newsletter covers each topic more comprehensively and consistently across runs.

**Why P1**: The core value proposition of the adaptive approach. Without this, the system continues to produce overlapping or gap-ridden research from pre-generated queries.

**Independent Test**: Configure a topic with `search_depth: "deep"` and `max_research_rounds: 3`. Run the pipeline. Inspect logs for `[AdaptiveResearch]` entries showing: (1) a planning step identifying key aspects, (2) each round's analysis identifying gaps, (3) each subsequent query being different from the previous and targeting an identified gap.

**Acceptance Scenarios**:
1. **Given** a topic with `search_depth: "deep"` and `max_research_rounds: 3`, **When** the research phase runs, **Then** the system executes a planning step followed by up to 3 search-analyze cycles, where each search query is informed by the previous round's analysis.
2. **Given** the first search round's analysis identifies 2 knowledge gaps, **When** the second round executes, **Then** its search query explicitly targets one of the identified gaps (verifiable in logs).
3. **Given** a deep-mode topic, **When** research completes, **Then** the final `research_{idx}_{provider}` state key contains a merged `SUMMARY:` and `SOURCES:` section covering findings from all rounds.

---

### US-ADR-02 -- Saturation Detection Reduces Waste (Priority: P1) MVP

**As an** Operator, **I want** the research loop to stop early when it detects that further searching would yield diminishing returns, **so that** I do not waste API calls and time when a topic is already well-covered.

**Why P1**: Without intelligent exit, the system always runs all configured rounds even when the first 2 rounds cover the topic fully. This wastes time and API quota.

**Independent Test**: Configure a narrow, well-documented topic with `max_research_rounds: 5`. Run the pipeline. Verify that the adaptive loop exits before round 5 with a log message mentioning saturation.

**Acceptance Scenarios**:
1. **Given** a topic with `max_research_rounds: 5`, **When** the AnalysisAgent reports `saturated: true` after round 2 (having completed >= 2 rounds), **Then** the loop exits immediately and logs the saturation event.
2. **Given** a topic where the AnalysisAgent reports `saturated: true` on round 0, **When** the orchestrator evaluates exit criteria, **Then** the saturation signal is overridden and the loop continues for at least one more round (safety minimum).
3. **Given** a topic where the AnalysisAgent reports empty `knowledge_gaps` after round 1, **When** the orchestrator evaluates exit criteria, **Then** the loop exits and logs that all aspects were covered.

---

### US-ADR-03 -- Reasoning Chain Transparency (Priority: P2)

**As an** Operator, **I want** to see the full reasoning chain (plan, per-round analysis, gap detection, query decisions) in the logs, **so that** I can debug research quality issues and understand why certain queries were chosen.

**Why P2**: Transparency is important for trust and debugging but does not affect the core research quality.

**Independent Test**: Run a deep-mode topic. Inspect logs for structured `[AdaptiveResearch]` messages at each stage: planning output, per-round query, per-round analysis summary, gap list, saturation decision, and exit reason.

**Acceptance Scenarios**:
1. **Given** a deep-mode topic running through the adaptive loop, **When** the planning phase completes, **Then** the log contains: `"[AdaptiveResearch] Topic {name}/{provider}: Plan - intent: {intent}, aspects: [{aspects}], initial query: {query}"`.
2. **Given** a search round completing, **When** the analysis phase finishes, **Then** the log contains: `"[AdaptiveResearch] Topic {name}/{provider} round {N}: {findings_summary}. Gaps: [{gaps}]. Saturated: {bool}"`.
3. **Given** the loop exiting, **When** the final merge completes, **Then** the log contains: `"[AdaptiveResearch] Topic {name}/{provider}: completed {N} rounds, {M} unique URLs, exit reason: {reason}"`.

---

### US-ADR-04 -- Search Budget Control (Priority: P2)

**As an** Operator, **I want** to configure a separate maximum number of search API calls per topic, **so that** I can control API costs independently from the number of reasoning rounds.

**Why P2**: Adds fine-grained cost control but the default (matching `max_research_rounds`) is sufficient for most use cases.

**Independent Test**: Set `max_research_rounds: 5` and `max_searches_per_topic: 3`. Run a deep-mode topic. Verify that at most 3 searches execute even though 5 reasoning rounds are allowed.

**Acceptance Scenarios**:
1. **Given** `max_research_rounds: 5` and `max_searches_per_topic: 3`, **When** deep-mode research runs, **Then** at most 3 search rounds execute.
2. **Given** `max_searches_per_topic` is omitted from config, **When** config loads, **Then** it defaults to the value of `max_research_rounds`.
3. **Given** `max_searches_per_topic: 0` (invalid), **When** config loads, **Then** `ConfigValidationError` is raised with message indicating minimum is 1.
4. **Given** `max_searches_per_topic: 16` (invalid), **When** config loads, **Then** `ConfigValidationError` is raised with message indicating maximum is 15.

### US-ADR-05 -- Backward Compatibility (Priority: P1) MVP

**As an** Operator, **I want** my existing `topics.yaml` with `max_research_rounds` and no new fields to continue working without changes, **so that** upgrading does not break my configuration.

**Why P1**: Breaking existing configs would prevent adoption.

**Independent Test**: Run the pipeline with an existing `topics.yaml` that has only `max_research_rounds: 3` and no `max_searches_per_topic`. Verify it completes successfully with 3 adaptive rounds per deep-mode topic.

**Acceptance Scenarios**:
1. **Given** an existing `topics.yaml` without `max_searches_per_topic`, **When** the pipeline runs, **Then** it completes successfully using `max_research_rounds` as both the round limit and search limit.
2. **Given** `max_research_rounds: 1`, **When** a deep-mode topic runs, **Then** exactly one search executes with no planning or analysis phase (single-round mode).
3. **Given** a standard-mode topic, **When** the research phase runs, **Then** behavior is identical to the current pipeline.

### Edge Cases

- **Invalid PlanningAgent output**: Falls back to original query as initial search query and default key aspects. No pipeline failure.
- **Invalid AnalysisAgent output**: Falls back to suffix-based query variant. Loop continues with next round.
- **All analysis rounds report saturated=true from round 0**: Safety minimum override ensures at least 2 rounds execute.
- **AnalysisAgent suggests a query identical to a previous round**: The orchestrator SHALL detect duplicate queries and append a distinguishing suffix (e.g., "latest developments", "recent changes") to avoid wasted searches.
- **Search round returns empty results**: Analysis still runs on the empty result. AnalysisAgent is expected to note the gap and suggest an alternative query angle.
- **Search budget exhausted before first analysis**: If `max_searches_per_topic: 1` and `max_research_rounds > 1`, exactly one search executes, analysis runs on it, but loop exits due to budget exhaustion (no second search).

## 6. User Flows

### Flow A: Adaptive Deep Research (per topic, per provider)

1. Pipeline build: for each topic with `search_depth: "deep"`, construct `DeepResearchOrchestrator` (unchanged from parent spec).
2. **Planning**: Orchestrator creates `PlanningAgent` (LlmAgent, no tools) and invokes it. PlanningAgent outputs plan JSON to `adaptive_plan_{idx}_{provider}` state key.
3. Orchestrator parses the plan. If valid: extract `initial_search_query` and `key_aspects`. If invalid: use original query and default aspects. Log the plan.
4. Initialize `AdaptiveContext` with plan data. Set `searches_remaining` from config.
5. **Round 0 - Search**: Orchestrator creates `DeepSearchRound` LlmAgent with `initial_search_query`. Invokes it. Reads output from `deep_research_latest_{idx}_{provider}`. Copies to `research_{idx}_{provider}_round_0`. Extracts URLs from output.
6. If `max_research_rounds == 1`: skip analysis, go to step 11. (Backward compat: single-round mode.)
7. **Round 0 - Analyze**: Orchestrator creates `AnalysisAgent` (LlmAgent, no tools) with context (plan, round 0 results, key aspects, 0 prior rounds). Invokes it. Reads output from `adaptive_analysis_{idx}_{provider}`.
8. Orchestrator parses analysis. Updates `AdaptiveContext.rounds[0]` with findings. Checks exit criteria:
   - If `saturated == true` AND `round_idx >= 1`: exit loop. (Round 0 saturation overridden.)
   - If `knowledge_gaps` is empty: exit loop.
   - If `round_idx + 1 >= max_research_rounds`: exit loop.
   - If `searches_done >= max_searches_per_topic`: exit loop.
   - Otherwise: extract `next_query`, continue.
9. **Rounds 1-N**: Repeat steps 5 (search with `next_query`), 7 (analyze with accumulated context), 8 (check exit) until exit criteria met.
10. For each subsequent round: log `"[AdaptiveResearch] Topic {name}/{provider} round {N}: query='{query}', findings='{summary}', gaps={gaps}, saturated={bool}"`.
11. **Merge**: Orchestrator reads all `research_{idx}_{provider}_round_{N}` keys. Merges using existing `_merge_rounds` logic (concatenate summaries, deduplicate URLs). Writes to `research_{idx}_{provider}`.
12. **Cleanup**: Orchestrator deletes all intermediate keys (`adaptive_plan_*`, `adaptive_analysis_*`, `adaptive_context_*`, `deep_research_latest_*`, `research_*_round_*`).
13. Yield final event: `"[AdaptiveResearch] Topic {name}/{provider}: completed {N} rounds, {M} unique URLs, exit reason: {reason}"`.
14. Control returns to main pipeline for verification and refinement.

### Flow B: Single-Round Deep Research (max_research_rounds == 1)

1. Orchestrator skips planning (no PlanningAgent invocation).
2. Orchestrator executes one `DeepSearchRound` with the original topic query.
3. Reads round output, writes directly to `research_{idx}_{provider}` (no merge needed).
4. Skips analysis. No `AdaptiveContext` created.
5. Yields completion event. Control returns to pipeline.
6. Behavior is identical to parent spec single-round mode.

### Flow C: Standard-Mode Research (unchanged)

Identical to parent spec. No adaptive loop. Single `LlmAgent` search per provider.

- All fields required.

- All fields required.
- `knowledge_gaps` may be empty list (indicates full coverage).

- When `saturated` is `false`: `next_query` must be a non-empty string. If missing, trigger fallback.
- `knowledge_gaps` capped at 5 items. If more: truncate to first 5.
- Entire output validated as JSON. Non-JSON output triggers fallback.

**DeepResearchOrchestrator behavior contract (replaces parent spec Section 4.3 contract)**:

1. If `max_rounds == 1`: skip to step 4 (single-round mode, no planning/analysis).
2. Create `PlanningAgent` LlmAgent sub-agent with planning prompt.
3. Invoke PlanningAgent via `async for event in planner.run_async(ctx)`. Parse output from `adaptive_plan_{idx}_{provider}` state key. If invalid: use fallback (original query + default aspects). Log plan.
4. Initialize tracking: `searches_done = 0`, `round_count = 0`, `accumulated_urls = set()`, `adaptive_context = {...}`.
5. For each round (0 to `max_rounds - 1`):
   a. Determine query: round 0 uses `initial_search_query` from plan (or original query in single-round mode). Subsequent rounds use `next_query` from previous analysis.
   b. Check for duplicate query: if query matches any previous round's query, append distinguishing suffix.
   c. Create `DeepSearchRound` LlmAgent sub-agent with current query.
   d. Invoke search agent. Read output from `deep_research_latest_{idx}_{provider}`. Copy to `research_{idx}_{provider}_round_{round_count}`. Extract URLs. Update `accumulated_urls`. Increment `searches_done`.
   e. Yield progress event: `"[AdaptiveResearch] Topic {name}/{provider} round {N}: searched '{query}', {X} new URLs, {Y} total"`.
   f. If `max_rounds == 1`: break (skip analysis in single-round mode).
   g. Create `AnalysisAgent` LlmAgent sub-agent with accumulated context, latest results, key aspects, remaining budget.
   h. Invoke analysis agent. Read output from `adaptive_analysis_{idx}_{provider}`. Parse JSON.
   i. If invalid JSON: use fallback (next suffix-based query, `saturated=false`).
   j. Update `AdaptiveContext.rounds` with findings_summary, knowledge_gaps, coverage_assessment, saturated, urls_found.
   k. Log analysis: `"[AdaptiveResearch] Topic {name}/{provider} round {N}: {findings_summary}. Gaps: [{gaps}]. Saturated: {saturated}"`.
   l. Check exit criteria:
      - `saturated == true` AND `round_count >= 1` (at least 2 rounds done): break with reason "saturation".
      - `len(knowledge_gaps) == 0`: break with reason "full_coverage".
      - `searches_done >= max_searches`: break with reason "search_budget_exhausted".
      - Next iteration would exceed `max_rounds`: break with reason "max_rounds_reached".
   m. If not exiting: extract `next_query` from analysis output for next round.
6. After loop: merge all round results into `research_{idx}_{provider}` using existing `_merge_rounds` logic.
7. Cleanup all intermediate state keys.
8. Yield final event with completion details and exit reason.

### 9.4 Key Design Decisions

**Decision ADR-1: Replace upfront query expansion with adaptive per-round reasoning**
- **Decision**: Remove `QueryExpanderAgent` and generate each round's search query adaptively based on the analysis of prior results.
- **Rationale**: Pre-generated queries cannot account for what was (or was not) found. Adaptive queries respond to actual gaps, reducing overlap and improving coverage. This mirrors how human researchers work -- they don't plan all queries before seeing any results.
- **Alternatives considered**: (a) Keep query expansion but reorder/filter based on analysis -- rejected because it still pre-commits to query angles, limiting adaptivity. (b) Use the LLM's tool-use capability to self-direct search -- rejected because `google_search` grounding is only available via `LlmAgent` tools, not directly callable, and self-directed search provides less control over the reasoning chain.
- **Consequences**: One additional LLM call per round (analysis) compared to the current approach. Total LLM calls per topic-provider: 1 (plan) + N (searches) + N-1 (analyses) vs current 1 (expansion) + N (searches). Net increase: N calls for a N-round execution. At gemini-2.5-flash pricing and prompt sizes, this adds approximately $0.002-0.005 per topic-provider.
- **Source**: ADK Custom Agents documentation, StoryFlowAgent pattern (https://google.github.io/adk-docs/agents/custom-agents/, consulted 2026-03-15).

**Decision ADR-2: LLM-driven saturation detection replaces URL counting**
- **Decision**: Replace the 15-URL threshold exit with `AnalysisAgent`-determined saturation.
- **Rationale**: URL count is a poor proxy for research quality. A topic may have 20 URLs but all from one angle, or 8 URLs covering all key aspects. The LLM can evaluate semantic coverage, which is the actual goal.
- **Alternatives considered**: (a) Hybrid: URL count + LLM judgment -- rejected because it adds complexity without clear benefit; the LLM naturally considers source volume in its assessment. (b) Embedding-based similarity of round outputs -- rejected due to dependency on embedding model and marginal benefit over direct LLM analysis.
- **Consequences**: Saturation detection quality depends on LLM reliability. Safety minimum (2 rounds) and max_rounds hard cap prevent pathological cases (always-saturated or never-saturated).

**Decision ADR-3: Single query per round (not multi-query)**
- **Decision**: Each Analysis round suggests exactly one `next_query`. The orchestrator executes one search per round.
- **Rationale**: Keeps the loop simple and predictable. Multi-query per round would complicate budget accounting, result attribution, and analysis prompt construction. The separate `max_searches_per_topic` config provides the needed budget flexibility.
- **Alternatives considered**: Allow 1-3 queries per analysis round -- rejected due to implementation complexity, harder-to-predict API costs, and the observation that focused single-query rounds with iterative feedback produce better results than batch queries.
- **Consequences**: `max_searches_per_topic` and `max_research_rounds` are effectively the same when set to equal values (1 search per round). When `max_searches_per_topic < max_research_rounds`, the search budget is the binding constraint.

**Decision ADR-4: No-tools for PlanningAgent and AnalysisAgent**
- **Decision**: PlanningAgent and AnalysisAgent are LlmAgents with no tools attached.
- **Rationale**: These agents perform pure reasoning -- interpreting queries, analyzing text, identifying gaps. They do not need search capabilities. Keeping them tool-free ensures they cannot accidentally trigger searches (which would consume the search budget) and makes their token cost predictable.
- **Consequences**: Planning and analysis quality depends entirely on the LLM's reasoning ability and the quality of the prompt context. gemini-2.5-flash has been validated as sufficient for this in the existing query expansion and refinement prompts.

## 10. Non-Functional Requirements

### 10.1 Performance

- **Planning phase**: Adds 1 LLM call (~2-3 seconds) per topic-provider combination. Runs once, not per round.
- **Analysis phase**: Adds 1 LLM call (~2-4 seconds) per round per topic-provider. The prompt grows slightly each round as `prior_rounds_summary` accumulates, but stays well within gemini-2.5-flash's context window.
- **Per-round overhead**: ~2-4 seconds for analysis (on top of ~10-15 seconds for search). Total per round: ~12-19 seconds.
- **Total 3-round execution estimate**: ~3s (plan) + 3 * ~15s (search+analyze) = ~48 seconds per topic-provider. Within the 3-5 minute target per SC-ADR-006.
- **Saturation early exit benefit**: When saturation triggers at round 2 (target: 30% of executions per SC-ADR-002), saves ~15s per topic-provider versus running all 3 rounds.
- **Single-round mode** (`max_research_rounds: 1`): No planning or analysis overhead. Performance identical to current implementation.

### 10.2 Security

- No new external attack surface.
- Planning and analysis prompts consume operator-controlled config values (topic name, query). No user-controlled input injection vector.
- Analysis prompts include search results from trusted LLM/API outputs. Same security posture as existing prompts.
- No new OWASP concerns beyond parent spec Section 10.2.

### 10.3 Scalability & Availability

- Analysis adds 1 LLM call per round per topic-provider. With 5 topics and 2 providers at 3 rounds each, worst case: 30 additional LLM calls (planning + analysis) per pipeline run. Within gemini-2.5-flash rate limits.
- `max_searches_per_topic` provides explicit budget control to prevent runaway API usage.

### 10.4 Accessibility

N/A -- no UI changes.

### 10.5 Observability

## 12. Constraints & Assumptions

### Constraints

- Google ADK `BaseAgent` custom agent pattern supports creating and invoking `LlmAgent` sub-agents dynamically within `_run_async_impl` (verified from ADK docs, StoryFlowAgent pattern).
- `google_search` grounding tool is only accessible via `LlmAgent` tools -- PlanningAgent and AnalysisAgent cannot use it directly.
- Analysis prompt context grows with each round. With 5 rounds of research, the analysis prompt may reach ~10-15K tokens. `gemini-2.5-flash` supports up to 1M token context, so this is well within limits.
- The adaptive loop is sequential within a single topic-provider (by design -- each round depends on the prior analysis). Cross-topic parallelism via `ParallelAgent` is preserved.

### Assumptions

- **A-ADR-1**: `gemini-2.5-flash` reliably produces valid JSON from both the planning and analysis prompts. Fallback logic is specified for all failure cases. Validation: the existing `QueryExpanderAgent` already uses this model for JSON output successfully.
- **A-ADR-2**: Analysis of search results via LLM produces meaningfully better query selection than pre-generated query variants. Validation: to be confirmed empirically during implementation.
- **A-ADR-3**: The additional LLM calls (1 planning + N-1 analysis per topic-provider) add acceptable latency (~2-4 seconds each) relative to search time (~10-15 seconds each). Validation: gemini-2.5-flash median response time for short prompts is 1-3 seconds per existing telemetry.
- **A-ADR-4**: The `prior_rounds_summary` context provided to the AnalysisAgent is sufficient for it to identify remaining gaps without seeing the full search results of all prior rounds. This is a token-efficiency assumption. Validation: the summary includes `findings_summary` and `knowledge_gaps` from each prior round, which captures the essential information.
- **A-ADR-5**: Saturation detection by the LLM is directionally correct (may have false positives/negatives but is better than the 15-URL threshold). The safety minimum (2 rounds) and hard cap (`max_research_rounds`) bound the worst cases.

## 13. Out of Scope

- **Cross-provider reasoning**: The adaptive loop runs independently per provider. No cross-provider gap analysis (e.g., "Google found X but Perplexity didn't"). This could be added as a future enhancement.
- **Multi-query per analysis round**: Each analysis suggests exactly one next query. Allowing multiple queries per round (e.g., parallel sub-searches) adds complexity without proven benefit.
- **Per-topic adaptive config**: Adaptive parameters (`max_research_rounds`, `max_searches_per_topic`) are global settings. Per-topic overrides are out of scope.
- **Reasoning chain in output**: The reasoning chain (planning + analyses) is logged but not included in the final `research_{idx}_{provider}` output or the newsletter. Future enhancement could add a "research methodology" section.
- **Adaptive saturation threshold config**: The saturation decision is entirely LLM-driven via the AnalysisAgent. No configurable numeric threshold. The LLM's judgment via the prompt guidelines is the mechanism.
- **Changes to DeepResearchRefinerAgent**: The source refinement phase (parent spec Section 4.4) is unchanged. It operates on the final merged output regardless of how it was produced.
- **Changes to standard mode**: Standard-mode research is completely untouched.

## 15. Glossary

| Term | Definition |
|------|-----------|
| Adaptive reasoning loop | The new research pattern where each search round's query is determined by analyzing previous results, rather than pre-generated |
| PlanningAgent | LlmAgent (no tools) that interprets the research query and produces an initial search strategy with key aspects |
| AnalysisAgent | LlmAgent (no tools) that evaluates search results, identifies knowledge gaps, determines saturation, and suggests the next query |
| Knowledge gap | A topic aspect or question that has not been adequately addressed by the searches conducted so far |
| Saturation | The condition where further searching would yield diminishing returns -- all key aspects are covered or new searches add no significant new information |
| AdaptiveContext | The accumulated state data structure tracking the planning output and per-round analysis results across the adaptive loop |
| Prior rounds summary | A condensed representation of all previous rounds' findings and gaps, passed to the AnalysisAgent to inform its analysis |
| Safety minimum | The configurable minimum number of rounds (`min_research_rounds`, default 2) that must execute before saturation can trigger early exit, preventing premature stop |
| Search budget | The `max_searches_per_topic` limit on total search API calls per topic-provider combination |
| Fan-out (deprecated) | The previous approach where all query variants were generated upfront before any search was executed |
